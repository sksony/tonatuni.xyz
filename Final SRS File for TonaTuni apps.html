Software Requirements Specification

1.Introduction

1.1 Purpose

The purpose of this document is to give a detailed description of the requirements for the “Tonatuni” It will state the purpose and complete declaration for the development of system. It will mainly describe what is required from the system in user’s terms. It will also explain system constraints, user interface and communication with other external applications. This document is primarily intended to be proposed to a customer for its approval and a reference for developing the first version of the system for the development team.

1.2 Document Conventions

  • Font family: Times New Roman
  • Font Size: 11
  • Line spacing: 1.15

We used BOLD characters to highlight many important words. We have also used bullets and numbering to enlist different ordered and unordered list.

1.3 Intended Audience and Reading Suggestions

This Software Requirements document is intended for:

  • Developers of this application who can review project and easily understand, on which features they should focus to meet the requirements (design and code the application – it sets the guidelines for future development).
  • Project testers: Some bugs can be easily found using software requirement specification. This way testing becomes more organized.
  • Project Managers to know well the project requirements.
  • End users of this application who want to know whether the application to be developed meet their expectations or not.

1.4 Product Scope

This system 'Tonatuni' is a dating application that allow people to meet and date over Internet. It allows users to register themselves on system’s database and create public profile. Users are shown cards of some relevant people and are allowed to chat with them

According to a study, almost half of the youngsters aged between 18-24 look for the dates on internet. This application will provide one platform to this audience in which they can easily communicate and find their perfect match easily. It’ll provide fast and easy to understand UI and, reliable and secure platform for its users.

1.5 References

The Web app of the project contains a brief description of the project, screenshots, links, and tutorials and of course it gives users the ability to download and contribute to the project in different ways, since it is an open source project.

2. Overall Description

2.1 Product Perspective

It is a very useful app for those who are seeking for date online. Although there are some similar apps available on the internet but these are expensive to use and require costly subscriptions. It is free to use and you just have to download free app for app store or visit web app through your browser.

2.2 Product Functions

The Product will mainly provide these functions to its users:-

user:

  • Register/Signup
  • Login
  • Displaying swipeable cards
  • Create/Edit Profile
  • Set Likes/Love
  • Create matches
  • Chat

Admin:

  • An admin can add , edit or delete a user account.
  • An admin can approve or dicard a user review

2.3 User Classes and Characteristics

This app is for all the users especially for the people aged between 17-40. Anyone with basic computer skills and internet skill can use this app because of its easy-to-use UI.

2.4 Operating Environment

  • Operating environment is as listed below.

    There will be two main operating environments:-

    • Web browsers
    • Android : Must support Android 4.4 and up versions

2.5 Design and Implementation Constraints

Internet connection is required. Without an internet connection, the users cannot log in to the system. So, they cannot interact with the system without internet.

Another constraint is data capacity. As we are not using our own data center, we are might face issue with capacity.

2.6 User Documentation

User documentation components are available on the website and app's user manual portion.

3.External Interface Requirements

3.1 User Interfaces

A user that opens the app first time should see the sign-up page with an option “I already have an account” which takes him to the login page.

If user is not first-time, he/she should be able to see home page, on which there will be two tabs. A first one is profile tab, which opens by default, where he/she can change/upload profile picture, edit his/her e-mail, phone number and password. Also users can set preferred language in their profile. There will be also a chat option where user can chat to matches on 'Tonatuni' app. This tab also shows likes/loves a user has received and some other details.

A second/right tab will be a search tab where user can search for dates. It includes a text bar where user can enter keywords for search and filter for search that allows user to select gender of dates to be searched. It also allows user to set preferred range of age (for example 18-22) for people to be searched. See figure for the interface for filter the mobile app (Web app filter will be similar).

3.2 Hardware Interfaces

The mobile and web application, both do not have any specific/designated hardware, therefore it does not have any direct hardware interfaces.

3.3 Software Interfaces

The mobile application uses camera application to capture photos for the profile picture and gallery/file manager application to upload pictures of user. This communication involves both reading and modification of data.

It also uses default keyboard of android phone to type login/signup information and search terms and some other fields as well. This communication requires “Allow access keyboard” permission from the admin/user.

3.4 Communications Interfaces

The different parts of the systems communicate with each other in different ways and they depend on each other. However the way in which these components communicate is not important for the system (specifically not in this document). So these communications are handled by OS of both android and web app.

4. System Features

4.1 Download Mobile Application

4.1.1 Description and Priority

A user will download mobile application from application store or from the link provided in website/web app. The priority of this requirement is very high.

4.1.2 Stimulus/Response Sequences

The user will open corresponding mobile application store to download the application.

4.2 User Registration

4.2.1 Description and Priority

This feature allows user to create new account on 'Tonatuni' app. This feature is of high priority and importance because user cannot proceed in app without registration.

4.2.2 Stimulus/Response Sequences

When user opens the app, a screen is showed where he/she is asked to enter name, age and gender. After that user will press next button. Then next screen appears where user is asked to choose username and password. The system will determine whether the username is already chosen or not, and also check whether password is 8 digit long or not. If these conditions are satisfied, the user will be show a confirmation message indicating that Registration is successful.

4.2.3 Functional Requirements

  • REQ-1: Add the information of user to the database.
  • REQ-2: Look in the data, to check whether the chosen username is already taken or not.

4.3 Login

4.3.1 Description and Priority

If user has registered, then he should be able to login to the 'Tonatuni' application. The login information will be stored on phone/computer and user will be automatically logged in next time.

Priority: High

4.3.2 Stimulus/Response Sequences

The login screen contains two input text boxes. One for entering username and other for entering password. User will press login button after entering username and password.

4.3.3 Functional Requirements

  • REQ 1-Once user has entered login information, the system should look into its database and check whether this username exists or not.
  • REQ 2-If username is found, the system will authenticate password as well. If password is also matched, User should be allowed access to his./her profile

In exception cases system should show relevant error message

4.4.Swipeable Cards

4.4.1 Description and Priority

Swipeable cards effect similar to the tinder cards effect. A card should display an image with several options (like, chat).

Priority: High

4.4.2 Stimulus/Response Sequences

A card should be shown to the user with a pictures of the relevant persons and contains several options, like/love, chat etc. A user decides whether he/she like looks of a person or not. If user like, he/she will press like button and it will enable chat option, otherwise they’ll swipe to the next card.

4.5 Set Loves/Likes

4.5.1 Description and Priority

When you like some person from displayed cards, you’ll hit like button. The person will know that you liked him/her. And this like will be added to his/her profile and a math will be made.

Priority:Medium

4.5.2 Stimulus/Response Sequences

A user should be shown Swipeable card, which displays the picture of a matched person. If user like it, he/she will press like button and that the liked person should be notified about this like.

4.5.3 Functional Requirements

  • REQ 1-A system should add like/loves to the profile of liked person and also update this information in database.

In exception cases system should show relevant error message

4.6 Chat

4.6.1 Description and Priority

Swipeable cards should have a chat option that allow users to chat with a person/match that they like.

Priority:Medium

4.6.2 Stimulus/Response Sequences

If a user likes a card, he should press chat button which should take him to the chat box/inbox, where he/she can communicate with that person.

4.6.3 Functional Requirements

  • REQ 1-The chat box should contain an option to video call and voice call.
  • REQ 2-An option should be given to user to share images in chat box.

In exception cases system should show relevant error message

4.7 Matching

4.7.1 Description and Priority

If user likes someone, he’ll press like/love button and this will create a match between them.

Priority:Medium

4.7.2 Functional Requirements

  • REQ 1-When user likes some person, system should allow user to chat with him/her.

In exception cases system should show relevant error message

4.8 User profile

4.8.1 Description and Priority

The user profile should display profile picture of the user, username, dob, and gender.

Priority:Medium

4.8.2 Stimulus/Response Sequences

When user successfully log in, he will be shown his profile with several options like change name, password, profile picture. The profile page will also show likes/loves that user has received.

4.8.3 Functional Requirements

  • REQ 1-A profile should display profile picture of the user and number of people that like him/her.
  • REQ 2-It should enable the user to change name, password and profile picture.

In exception cases system should show relevant error message

5.Other Non-functional Requirements

5.1 Performance Requirements

The app should be interactive and involved delays must be minimum .So in every action/response of this app, there should be no immediate delays. The system should be highly responsive to the actions involved. Login process for good internet connection should not take more than 4 seconds. The uploading of pictures should be fast and immediate. The chat between the users should be fast and reliable.

5.2 Safety Requirements

In case of a server breakdown, the last checkpoint will be saved automatically.

5.3 Security Requirements

The communication between server and the system should be highly secured and private. The login requests to the server should be end to end encrypted, so others cannot extract usernames/password during process. If someone without an account tries to login, the system should not allow him to do so, a user’s IP should not be able to login after three failed login attempts. If a user wants to create an account and the desired user name is occupied, the user should be asked to choose a different user name.

5.4 Software Quality Attributes

The system should consider these characteristics:-

  • Adaptability
  • Availability(Availability of the system when it is used)
  • Correctness
  • Reliability
  • Maintainability: The app should be easy to extend/modify. The code should be written in that way that it favors implementation of new functions.
  • Portability : The mobile app should be portable with different versions of android

5.5 Business Rules

  • A User cannot check other users' credentials
  • Users confidentiality must be kept.

6. Other Requirements

The website & App will require to follow the legal requirements while being used such as US Digital Millennium Copyright Act. The project is sustainable and can be reused for similar purposes such as for online interaction purposes.
