4.3 Login

4.3.1 Description and Priority

If user has registered, then he should be able to login to the 'Tonatuni' application. The login information will be stored on phone/computer and user will be automatically logged in next time.

Priority: High

4.3.2 Stimulus/Response Sequences

The login screen contains two input text boxes. One for entering username and other for entering password. User will press login button after entering username and password.

4.3.3 Functional Requirements

  • REQ 1-Once user has entered login information, the system should look into its database and check whether this username exists or not.
  • REQ 2-If username is found, the system will authenticate password as well. If password is also matched, User should be allowed access to his./her profile

In exception cases system should show relevant error message

4.4.Swipeable Cards

4.4.1 Description and Priority

Swipeable cards effect similar to the tinder cards effect. A card should display an image with several options (like, chat).

Priority: High

4.4.2 Stimulus/Response Sequences

A card should be shown to the user with a pictures of the relevant persons and contains several options, like/love, chat etc. A user decides whether he/she like looks of a person or not. If user like, he/she will press like button and it will enable chat option, otherwise they’ll swipe to the next card.

4.5 Set Loves/Likes

4.5.1 Description and Priority

When you like some person from displayed cards, you’ll hit like button. The person will know that you liked him/her. And this like will be added to his/her profile and a math will be made.

Priority:Medium

4.5.2 Stimulus/Response Sequences

A user should be shown Swipeable card, which displays the picture of a matched person. If user like it, he/she will press like button and that the liked person should be notified about this like.

4.5.3 Functional Requirements

  • REQ 1-A system should add like/loves to the profile of liked person and also update this information in database.

In exception cases system should show relevant error message

4.6 Chat

4.6.1 Description and Priority

Swipeable cards should have a chat option that allow users to chat with a person/match that they like.

Priority:Medium

4.6.2 Stimulus/Response Sequences

If a user likes a card, he should press chat button which should take him to the chat box/inbox, where he/she can communicate with that person.

4.6.3 Functional Requirements

  • REQ 1-The chat box should contain an option to video call and voice call.
  • REQ 2-An option should be given to user to share images in chat box.

In exception cases system should show relevant error message

4.7 Matching

4.7.1 Description and Priority

If user likes someone, he’ll press like/love button and this will create a match between them.

Priority:Medium

4.7.2 Functional Requirements

  • REQ 1-When user likes some person, system should allow user to chat with him/her.

In exception cases system should show relevant error message

4.8 User profile

4.8.1 Description and Priority

The user profile should display profile picture of the user, username, dob, and gender.

Priority:Medium

4.8.2 Stimulus/Response Sequences

When user successfully log in, he will be shown his profile with several options like change name, password, profile picture. The profile page will also show likes/loves that user has received.

4.8.3 Functional Requirements

  • REQ 1-A profile should display profile picture of the user and number of people that like him/her.
  • REQ 2-It should enable the user to change name, password and profile picture.

In exception cases system should show relevant error message

5.Other Non-functional Requirements

5.1 Performance Requirements

The app should be interactive and involved delays must be minimum .So in every action/response of this app, there should be no immediate delays. The system should be highly responsive to the actions involved. Login process for good internet connection should not take more than 4 seconds. The uploading of pictures should be fast and immediate. The chat between the users should be fast and reliable.

5.2 Safety Requirements

In case of a server breakdown, the last checkpoint will be saved automatically.

5.3 Security Requirements

The communication between server and the system should be highly secured and private. The login requests to the server should be end to end encrypted, so others cannot extract usernames/password during process. If someone without an account tries to login, the system should not allow him to do so, a user’s IP should not be able to login after three failed login attempts. If a user wants to create an account and the desired user name is occupied, the user should be asked to choose a different user name.

5.4 Software Quality Attributes

The system should consider these characteristics:-

  • Adaptability
  • Availability(Availability of the system when it is used)
  • Correctness
  • Reliability
  • Maintainability: The app should be easy to extend/modify. The code should be written in that way that it favors implementation of new functions.
  • Portability : The mobile app should be portable with different versions of android

5.5 Business Rules

  • A User cannot check other users' credentials
  • Users confidentiality must be kept.

6. Other Requirements

The website & App will require to follow the legal requirements while being used such as US Digital Millennium Copyright Act. The project is sustainable and can be reused for similar purposes such as for online interaction purposes.
